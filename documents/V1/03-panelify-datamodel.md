_Data Model Document_

# **1\. Overview**

Panelify uses a deliberately minimal data model. The principle is: store as little as possible in a database. Content lives in GitHub. Images live in GitHub. Sessions live in Supabase Auth. The only custom database table stores the connection metadata between a user and their repo.

WHERE DATA LIVES IN PANELIFY

User credentials → Supabase Auth (managed, encrypted)

Site metadata → Supabase — sites table (1 row per site)

Site content → GitHub — content.json in the user's repo

Images → GitHub — /public/images in the user's repo

Unpublished changes → Browser React state (not persisted)

Publish history → GitHub commit history (free, automatic)

Backup → GitHub — panelify-backup branch

# **2\. Supabase — auth.users (managed by Supabase)**

This table is managed entirely by Supabase Auth. Panelify does not create or modify it. It is referenced by the sites table via foreign key.

| **Column** | **Type** | **Source** | **Notes** |
| --- | --- | --- | --- |
| id  | UUID | Supabase | Primary key — referenced by sites.user_id |
| email | TEXT | GitHub OAuth | User's GitHub email |
| raw_user_meta_data | JSONB | GitHub OAuth | Contains github username, avatar_url |
| created_at | TIMESTAMPTZ | Supabase | Account creation timestamp |
| last_sign_in_at | TIMESTAMPTZ | Supabase | Last login timestamp |

# **3\. Supabase — sites (custom table)**

One row per connected site per user. In V1 each user has exactly one site. This table stores everything Panelify needs to reconnect to the repo on subsequent logins. The platform column was added in v1.1 and is required.

| **Column** | **Type** | **Nullable** | **Description** |
| --- | --- | --- | --- |
| id  | UUID | NO  | Primary key — auto-generated |
| user_id | UUID | NO  | Foreign key to auth.users.id — row level security uses this |
| repo_name | TEXT | NO  | Short repo name e.g. my-agency-site |
| repo_owner | TEXT | NO  | GitHub username of repo owner e.g. devuser |
| repo_full_name | TEXT | NO  | Full name e.g. devuser/my-agency-site — unique constraint |
| default_branch | TEXT | NO  | Branch Panelify commits to — defaults to main |
| platform | TEXT | NO  | Hosting platform — 'vercel', 'netlify', or 'unknown'. Set on repo connection. Used on every publish. Default: 'unknown' |
| vercel_url | TEXT | YES | Auto-detected Vercel URL e.g. my-agency-site.vercel.app |
| last_published_at | TIMESTAMPTZ | YES | Timestamp of last successful publish |
| created_at | TIMESTAMPTZ | NO  | When the site was first connected to Panelify |

## **SQL — Full Table Definition**

\-- Full table definition (v1.1 — includes platform column)

CREATE TABLE sites (

id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,

repo_name TEXT NOT NULL,

repo_owner TEXT NOT NULL,

repo_full_name TEXT NOT NULL,

default_branch TEXT NOT NULL DEFAULT 'main',

platform TEXT NOT NULL DEFAULT 'unknown', -- NEW in v1.1

vercel_url TEXT,

last_published_at TIMESTAMPTZ,

created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

UNIQUE(user_id, repo_full_name)

);

\-- Row Level Security — users can only see their own sites

ALTER TABLE sites ENABLE ROW LEVEL SECURITY;

CREATE POLICY sites_select ON sites FOR SELECT USING (auth.uid() = user_id);

CREATE POLICY sites_insert ON sites FOR INSERT WITH CHECK (auth.uid() = user_id);

CREATE POLICY sites_update ON sites FOR UPDATE USING (auth.uid() = user_id);

CREATE POLICY sites_delete ON sites FOR DELETE USING (auth.uid() = user_id);

# **4\. GitHub — content.json**

This file is the actual content database for the user's site. It lives in their GitHub repo at the root level. Panelify creates it during setup and reads/writes it on every publish. The structure mirrors the site's sections and components.

// content.json — example structure

{

"hero": {

"title": "We Build Amazing Products",

"subtitle": "A passionate team of developers...",

"image": "/images/hero.jpg",

"cta": "Get Started"

},

"about": {

"title": "Who We Are",

"body": "We are a team of 10 designers...",

"image": "/images/team.jpg"

},

"services": {

"title": "What We Do",

"subtitle": "We help startups build faster"

},

"footer": {

"copyright": "© 2026 My Agency. All rights reserved.",

"email": "hello@myagency.com"

}

}

## **content.json Rules**

- Keys are snake_case section names matching the component filename without .jsx
- Nested keys are field IDs generated by the AI scanner
- Image values are paths relative to /public — e.g. /images/hero.jpg
- The file is always valid JSON — Panelify validates before committing
- Max recommended size is 50KB — sufficient for any landing page

# **6\. Browser State (React)**

Unpublished changes live in React state only. They are never written to a database or local storage. If the owner closes the browser before publishing, the changes are lost. This is an intentional V1 simplification.

// Pending changes — stored in React state on the dashboard

interface PendingChanges {

\[section: string\]: {

\[field_id: string\]: string; // new value

}

}

// Example state after editing hero title and about image:

{

hero: { title: 'We Build Digital Dreams' }, // changed

about: { image: '/images/new-team.jpg' } // changed

}

// On Publish: this state is merged with content.json

// and committed to GitHub as a single update.

# **7\. Data Lifecycle**

| **Event** | **Data Created** | **Where Stored** | **Deleted When** |
| --- | --- | --- | --- |
| User signs up | auth.users row, session token | Supabase Auth | User deletes account |
| User connects repo | sites row (with platform) | Supabase sites table | User disconnects site |
| Panelify setup runs | content.json, backup branch, refactored JSX | GitHub repo | User deletes from repo |
| User edits section | Pending changes object | Browser React state | Page refresh or publish |
| User publishes | GitHub commit + updated content.json | GitHub repo | Never — Git is permanent |
| Image uploaded | Image file in /public/images | GitHub repo | User deletes from repo |
